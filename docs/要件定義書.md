## 1. システム概要

### 1.1 目的

DJや外注スタッフから Gmail に届く請求書 PDF を自動で検知・解析し、Google スプレッドシートへ転記する作業を完全自動化する。  
Google Gemini API による AI 解析で PDF 内容を読み取り、手作業による転記ミス・工数を排除することを目的とする。

### 1.2 解決する課題

- PDF請求書の手動確認・転記作業による工数とミスの発生
- 月次集計や年度切り替え時のシート管理の煩雑さ
- 締め切り超過請求書の見落としリスク
- 処理済み／未処理メールの管理漏れ

### 1.3 システム名

**請求書自動転記システム**（内部コード: DJ_INVOICE_THP）

## 4. 機能要件

### 4.1 メール監視・検索（FR-01）

- 毎時1回、時間ベーストリガーにより `checkAndProcessInvoices` を実行する
- Gmail を以下の条件で検索し、未処理のスレッドを最大 `MAX_PROCESS_PER_RUN`（10件）取得する
  - PDF添付ファイルあり（`has:attachment filename:pdf`）
  - ファイル名に「請求書」を含む（`filename:請求書`）
  - 処理済みラベル未付与（`-label:請求書処理済み`）

### 4.2 PDF添付ファイルの抽出（FR-02）

- メールの添付ファイルから PDF のみを抽出する
  - 条件: ContentType が `application/pdf` またはファイル名末尾が `.pdf`
  - かつファイル名に「請求書」を含む
- 対象PDFが存在しない場合は処理をスキップする（処理済みラベルは付与しない）

### 4.3 Gemini API によるPDF解析（FR-03）

- PDF を Base64 エンコードして Gemini API（`gemini-2.5-flash`）へ送信する
- 明細を **月ごとに集計** して JSON 配列形式で返却させる
- 複数月にまたがる請求書は月ごとに別オブジェクトとして返却させる

#### 抽出フィールド（月単位）

| フィールド | 型 | 説明 |
|-----------|-----|------|
| `name` | string | 請求者の本名 |
| `stage_name` | string | 請求者の芸名（なければ空文字） |
| `amount` | number | その月の明細金額の合計（源泉徴収後の金額を優先） |
| `invoice_date` | string | 請求月（「4月」のような M月形式） |
| `content` | string | 件名または品目名（複数の場合はカンマ連結） |
| `payment_due` | string | 空文字（システム側で計算するため） |

#### 芸名の抽出ルール

- 請求者名の直後に改行で別の名前が記載されている場合、1行目を `name`、2行目を `stage_name` に入れる
- 例: 「山田太郎\nDJ Taro」→ `name: "山田太郎", stage_name: "DJ Taro"`
- 芸名の記載がない場合は `stage_name` を空文字にする

### 4.4 スプレッドシート転記（FR-04）

- Gemini API の解析結果（月ごとの配列）をループし、各月のデータを1行ずつ転記する
- 転記先シートは受信日に基づいて決定する（下記ロジック参照）

#### スプレッドシート・シート決定ロジック

```
受信月が4月（新年度開始月）?
  ├─ YES → FOLDER_ID のフォルダに新スプレッドシートを作成
  │         ファイル名: 「YYYY年度 請求書管理」
  │         同名ファイルが存在する場合は既存ファイルを使用
  └─ NO  → SPREADSHEET_ID のスプレッドシートを使用
             該当月のシートが存在するか確認
               ├─ なし → 新シートを作成してヘッダー行を挿入
               └─ あり → 既存シートを使用
```

#### シート名の形式

```
YYYY.M (731締め)
例: 2026.5 (731締め)
```

#### ヘッダー行（新シート作成時に1行目へ挿入）

```
到着月日 | 名前 | 芸名 | 金額 | 請求月日 | 内容 | 支払い予定
```

#### 転記フィールドのマッピング

| 列 | 項目 | 値 |
|----|------|-----|
| B | 到着月日 | メール受信日（M/d 形式で表示） |
| C | 名前 | Gemini抽出の `name` |
| D | 芸名 | Gemini抽出の `stage_name` |
| E | 金額 | Gemini抽出の `amount`（数値） |
| F | 請求月日 | Gemini抽出の `invoice_date` |
| G | 内容 | Gemini抽出の `content` |
| H | 支払い予定 | 受信日の翌々月末（例: 7/31） |

#### 支払い予定の計算ロジック

- 受信日の翌々月末日を算出してH列に記録する
- 例: 受信日 2026/05/13 → 支払い予定 7/31

### 4.5 締め切り超過チェック（FR-05）

- 受信日の「日」が `CONFIG.CLOSING_DAY`（5日）を超えている場合に締め切り超過と判定する
- H列（支払い予定）の現在の値の末尾に `⚠️締切超過` を追記する
- 超過通知メールを `CONFIG.ERROR_NOTIFY_EMAIL` へ送信する
  - 件名: `[要確認] 締め切り超過の請求書を受信しました`
  - 本文: 受信日・送信者・金額・転記先シート名を含む

### 4.6 二重処理防止（FR-06）

- 処理が完了したメールのスレッドに Gmailラベル `請求書処理済み` を付与する
- 次回の監視時、ラベル付きスレッドは検索条件から除外されスキップされる
- ラベルが存在しない場合は `GmailApp.createLabel()` で自動作成する

### 4.7 トリガー設定（FR-07）

- `setupTrigger` 関数を手動で一度だけ実行することで毎時トリガーを設定する
- 既存の同名トリガーは重複防止のため自動削除する
- トリガー設定後は `checkAndProcessInvoices` が毎時自動実行される

### 4.8 デバッグ・テスト機能（FR-08）

| 関数名 | 用途 |
|--------|------|
| `runOnce` | 未処理メールを1件だけ取得して転記まで実行（動作確認用） |
| `debugGmailSearch` | Gmail検索状態を段階的に確認（メールが見つからない場合の原因特定） |
| `cleanupSheet` | シートの3行目以降のデータを全削除（テスト後のクリーンアップ用） |
| `testProcessInvoice` | 5つのシナリオテストを一括実行（下記参照） |

#### テストシナリオ

| No. | テスト内容 | 期待結果 |
|-----|-----------|---------|
| 1 | 通常の請求書メール（5日以内受信） | 正常転記、超過フラグなし |
| 2 | 締め切り超過メール（6日以降受信） | H列に ⚠️締切超過 が追記される |
| 3 | 新しい月の初回請求書 | 新シートが自動作成される |
| 4 | 4月の請求書（新年度） | 新スプレッドシートが自動作成される |
| 5 | PDF解析失敗 | エラー行が記録され通知メールが送信される |

## 5. 非機能要件

### 5.1 可用性

- GAS Webアプリは Google のインフラに依存するため、独自の冗長化は不要
- Google サービスの障害時は業務停止を許容する

### 5.2 パフォーマンス

- 1回のトリガー実行あたりの最大処理件数は **10件**（`MAX_PROCESS_PER_RUN = 10`）とする
  - GAS の実行時間制限（6分）を超えないための措置
- 未処理メールが10件を超える場合は次回実行時に続きを処理する

### 5.3 スケーラビリティ

- 外注スタッフ数: 最大 **50名程度** を想定
- 月次データ行数: 最大 **200行/月程度** を想定
- 同時処理対象メール: 最大 **10件/時間** を想定

### 5.4 保守性

- GASコードは clasp で Git 管理する
- すべての関数に日本語コメントを付与し、保守担当者が理解できる状態を維持する
- エラーハンドリングは try-catch で行い、エラー内容を `Logger.log()` にも出力する

## 6. データ設計

### 6.1 CONFIG 設定定数

| キー | 型 | 説明 |
|------|----|------|
| `SPREADSHEET_ID` | string | 現在使用中のスプレッドシートID |
| `FOLDER_ID` | string | 新年度ファイルの保存先 Google Drive フォルダID |
| `GEMINI_API_KEY` | string | Gemini APIキー |
| `SHEET_SUFFIX` | string | シート名のサフィックス（デフォルト: `731締め`） |
| `CLOSING_DAY` | number | 締め切り日（デフォルト: `5`） |
| `ERROR_NOTIFY_EMAIL` | string | エラー通知先メールアドレス |
| `PROCESSED_LABEL` | string | 処理済みGmailラベル名（デフォルト: `請求書処理済み`） |
| `FISCAL_YEAR_START` | number | 年度開始月（デフォルト: `4`） |

### 6.2 Gemini API レスポンス（JSON配列）

```json
[
  {
    "name": "山田太郎",
    "stage_name": "DJ TARO",
    "amount": 115829,
    "invoice_date": "4月",
    "content": "DJ出演料, 交通費",
    "payment_due": ""
  },
  {
    "name": "山田太郎",
    "stage_name": "DJ TARO",
    "amount": 80000,
    "invoice_date": "5月",
    "content": "DJ出演料",
    "payment_due": ""
  }
]
```

### 6.3 スプレッドシート転記フォーマット

#### 月次シート（例: `2026.5 (731締め)`）

| 行 | B列（到着月日） | C列（名前） | D列（芸名） | E列（金額） | F列（請求月日） | G列（内容） | H列（支払い予定） |
|----|---------------|------------|------------|------------|---------------|------------|-----------------|
| 1 | 到着月日 | 名前 | 芸名 | 金額 | 請求月日 | 内容 | 支払い予定 |
| 2〜 | 5/13 | 西尾 拓 | DJ BANKO | 115829 | 4月 | ジラフARC | 7/31 |

> **注意**: B列を上から走査して最初の空白セルの直前行を「最終データ行」として扱うため、途中に空白行がないことが前提。

### 6.4 エラー行フォーマット

PDF解析失敗時は以下の形式でエラー行を記録する：

| B列（到着月日） | C列（送信者） | D列（芸名） | E列（金額） | F列（請求月日） | G列（内容） | H列（支払い予定） |
|---------------|-------------|------------|------------|---------------|------------|-----------------|
| 受信日 | 送信者メールアドレス | （空） | 要確認 | - | エラーメッセージ（100文字まで） | - |

## 8. エラーハンドリング

### 8.1 PDF解析失敗時

- 該当行のE列（金額）に `要確認` と記入してシートに追記する
- エラー通知メールを送信する
  - 件名: `[エラー] 請求書の自動転記に失敗しました`
  - 本文: 受信日・送信者・エラー内容を含む

### 8.2 必須項目の取得失敗時

- 取得できた項目のみ転記し、取得できなかった項目は `-` で埋める（`sanitizeInvoiceData_` で処理）
- エラー通知メールを送信する

### 8.3 PDF以外の添付ファイルのみの場合

- 処理をスキップする（処理済みラベルは付与しない）

### 8.4 Gemini API エラー時

- HTTP ステータスコードが 200 以外の場合は例外をスローし、呼び出し元で catch する
- JSON 解析エラー時も例外をスローし、エラー行として記録する

### 8.5 メイン処理レベルのエラー

- `checkAndProcessInvoices` 全体を try-catch で囲み、予期せぬエラーをキャッチする
- エラー内容とスタックトレースを Logger.log に出力し、通知メールを送信する

## 9. セキュリティ要件

| 要件 | 内容 |
|------|------|
| APIキー管理 | `GEMINI_API_KEY` は `Code.gs` の CONFIG オブジェクトで管理（**本番運用前にスクリプトプロパティへの移行を推奨**） |
| アクセス制御 | GAS は OAuth2.0 によるアカウント認証済みユーザーのみが実行可能 |
| PDF送信 | Gmail に届いたメールのみを対象とし、外部から任意のPDFを注入できない設計 |
| 転記先制限 | 設定済みの SPREADSHEET_ID / FOLDER_ID のみに書き込み |

> **注意**: 現在 CONFIG に `GEMINI_API_KEY` をハードコードしているため、コードリポジトリへのプッシュ前に GAS スクリプトプロパティ（`PropertiesService.getScriptProperties()`）への移行を強く推奨する。

## 10. 運用・保守要件

### 10.1 初回セットアップ手順

1. **Gemini APIキーの取得**  
   [Google AI Studio](https://aistudio.google.com/) にてAPIキーを発行する

2. **環境変数の設定**  
   `Code.gs` の CONFIG オブジェクトを自身の環境に合わせて書き換える

3. **デプロイ**  
   clasp または GAS エディタに直接コードを貼り付けて保存する  
   ```bash
   clasp push
   ```

4. **トリガーの設定**  
   GAS エディタから `setupTrigger` 関数を一度だけ手動実行する  
   → 以降は毎時自動で `checkAndProcessInvoices` が実行される

### 10.2 定期実行トリガー

| 関数名 | 実行タイミング | 説明 |
|--------|--------------|------|
| `checkAndProcessInvoices` | 毎時（1時間ごと） | Gmail監視・PDF解析・スプレッドシート転記 |

### 10.3 コードの更新・デプロイ方法

```bash
# GASへアップロード
clasp push

# GASからダウンロード
clasp pull
```

### 10.4 処理済みラベルの管理

- Gmail の `請求書処理済み` ラベルで処理済みメールを識別する
- 再処理が必要な場合はラベルを手動で外す

### 10.5 新年度切り替えの運用

- 4月1日以降に届いた請求書は自動的に新年度スプレッドシートへ転記される
- 新スプレッドシートは FOLDER_ID で指定したフォルダに `YYYY年度 請求書管理` という名前で自動作成される
- 同名ファイルが既に存在する場合は既存ファイルを使用する（重複作成なし）

## 11. 制約・前提条件

### 11.1 外部サービス依存

- Google アカウント（オーナー）が有効であること
- Google スプレッドシート、Google Drive、Gmail の利用制限に収まること
- Gemini API の利用制限に収まること（無料プランは1日あたりのリクエスト数に上限あり）

### 11.2 GAS 実行制限（Google の制約）

| 制限 | 上限 |
|------|------|
| スクリプト実行時間 | 6分 / 実行 |
| 1日のトリガー実行回数 | 20回 / ユーザー |
| メール送信数 | 100通 / 日（無料アカウント） |

### 11.3 対象ファイルの命名規則

- PDFのファイル名に「**請求書**」という文字列が含まれている必要がある
- ファイル名に「請求書」が含まれていない PDF は処理対象外となる

### 11.4 データ修正について

- 一度転記されたデータの修正はスプレッドシートを直接編集することで対応する
- GASからの再転記機能は実装していない

### 11.5 複数月にまたがる請求書

- 1枚のPDFに複数月の明細が含まれる場合、月ごとに集計して別々の行として転記する
- この際、どちらも受信日が同じであるため、転記先シートは受信日の月に統一される

### 11.6 その他前提

- 外注スタッフは請求書PDFを Gmail で送付すること
- PDFのファイル名に「請求書」が含まれること
- 管理者は Google アカウントへのアクセス権を持つこと
